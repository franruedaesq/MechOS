# MechOS Architecture: Brain vs. Kernel

MechOS enforces a strict separation of concerns to solve the problem of AI hallucination and physical safety in robotics. The system is split into two primary components: the Brain (AI) and the Kernel (MechOS).

## 1. What decision does the LLM take and how?

The LLM (Brain) is responsible for high-level cognitive decisions. It operates within the `AgentLoop` (the OODA loop - Observe, Orient, Decide, Act, Gatekeep).

During the **Decide** phase, the system prompts the LLM with the current physical state of the robot (position, heading, velocity, obstacle detection via a 3D Octree, and recent memories). The LLM is forced to output ONLY a single JSON object that conforms to the `HardwareIntent` schema.

A `HardwareIntent` represents a discrete physical action or communication request. The possible decisions (intents) the LLM can make are strictly defined in `mechos-types` as:
* `MoveEndEffector { x, y, z }` - High-level 3D coordinate movement.
* `Drive { linear_velocity, angular_velocity }` - Differential drive command.
* `TriggerRelay { relay_id, state }` - Discrete hardware action.
* `AskHuman { question, context_image_id }` - Human-In-The-Loop (HITL) request for guidance.
* `MessagePeer { target_robot_id, message }` - Direct message to another robot.
* `BroadcastFleet { message }` - Broadcast to the entire fleet.
* `PostTask { title, description }` - Post a task to the shared Fleet Task Board.

## 2. How are these decisions translated into physical actions?

Before any `HardwareIntent` generated by the LLM can be executed, it must pass through the **Gatekeep** phase. This is the core responsibility of the MechOS Kernel, specifically the `KernelGate`.

The `KernelGate` performs two critical checks:
1. **Capability Check (`CapabilityManager`)**: Ensures the agent has the correct permissions (capabilities) to perform the requested action (e.g., `HardwareInvoke("drive_base")` is required for a `Drive` intent).
2. **Physical Invariant Check (`StateVerifier`)**: Validates the intent against a set of registered safety rules. For example, the `SpeedCapRule` will reject a `Drive` command if the requested linear or angular velocity exceeds the maximum safe limits. The `EndEffectorWorkspaceRule` rejects any `MoveEndEffector` command that attempts to move outside the safe operational workspace.

If any check fails, a `MechError::HardwareFault` or `MechError::Unauthorized` is raised, and the intent is rejected before ever reaching the hardware.

## 3. What is the output of this translation and where are they sent?

If the `HardwareIntent` passes the `KernelGate`, it is published to the `EventBus` (specifically on the `HardwareCommands` topic) as an `EventPayload::AgentThought`.

The output is then received by various middleware adapters or the Hardware Abstraction Layer (HAL), depending on the system configuration:

* **Hardware Abstraction Layer (`mechos-hal`)**: The `HardwareRegistry` receives the intent and dispatches it to the specific registered physical actuators or relays. For example, a `Drive` intent is decomposed into left and right wheel targets based on the robot's kinematics and sent to the corresponding motor drivers. A `MoveEndEffector` intent is sent to an Inverse Kinematics (IK) adapter to resolve joint angles.
* **ROS 2 Integration (`mechos-middleware::Ros2Adapter`)**: In a ROS 2 environment, the adapter translates the `HardwareIntent` into standard ROS 2 message frames (e.g., `geometry_msgs/msg/Twist` for `Drive`, or `std_msgs/msg/String` for fleet communications) and publishes them to the appropriate ROS topics (e.g., `/cmd_vel`).
* **Digital Twin Simulation (`mechos-middleware::DashboardSimAdapter`)**: For simulation or the frontend dashboard, the intent is translated into a JSON payload and forwarded to the dashboard via WebSockets to animate the virtual robot.

## 4. Is all this working correctly? Is it possible?

Yes, this architecture is fully implemented and operational within the MechOS codebase.

The code strictly enforces this separation:
* The LLM (`LlmDriver`) only returns strings which are strictly parsed into the `HardwareIntent` enum.
* The `AgentLoop` guarantees that `self.gate.authorize_and_verify()` is called *before* the intent is published to the `EventBus`.
* The `StateVerifier` and `CapabilityManager` ensure that no unsafe or unauthorized command can proceed past the Kernel layer.
* The tests within `crates/mechos-runtime/src/agent_loop.rs` and `crates/mechos-kernel/src/state_verifier.rs` confirm that invalid intents (e.g., exceeding speed caps, moving outside the workspace, or lacking capabilities) are successfully intercepted and rejected, while valid intents are correctly dispatched.
* The `Ros2Adapter`, `DashboardSimAdapter`, and `HardwareRegistry` all contain robust implementations for parsing the approved `HardwareIntent` and routing it to the appropriate physical or simulated endpoints.
